# DC Sync Probe — Specification

## Problem

Our team cannot access the production Salesforce environment to debug user sync issues. We can only observe production behavior through log files that capture the exact changes the app attempted to sync.

These log files come as two JSON files per sync session:
- **CreateChanges** — new records the app tried to create
- **UpdateChanges** — field modifications the app tried to apply

Both files also contain an `initialDCJSON` key: the full meeting snapshot as it was downloaded from production SF at the time of the sync.

We need a tool that **replays these production sync operations against a staging environment** so we can observe and debug the results.

---

## Capabilities

The tool provides three independent operations that can be composed together.

### Capability 1: Clone

**Make a staging meeting's data match a given DCJSON.**

**Input:** A source DCJSON (e.g. `initialDCJSON` from production logs) + a target staging meeting (found by search).

**What it does:**
- Sanitizes the source DCJSON so it's valid for the target meeting (PII replaced, IDs remapped, sync metadata stripped, staging meeting identifiers applied)
- Diffs the sanitized DCJSON against the target meeting's current state to produce changes
- Syncs those changes (creates first, then updates)

**Verification:** After syncing, re-pulls the meeting and compares it against the sanitized DCJSON. Produces a report of matched/mismatched/missing fields. Allowed differences:
- IDs (always regenerated by SF)
- Fields that don't sync to Salesforce (internal-only fields, non-syncable metadata)
- PII fields that were replaced with synthetic values

**Output:**
- A staging meeting whose syncable data matches the source DCJSON
- An **ID map** (`source ID → cloned ID`) for every repeater item created during cloning. This map is persisted to the dump directory and is the primary mechanism for ID remapping in Apply Changes.
- A verification report confirming the clone succeeded

### Capability 2: Apply Changes

**Replay a set of change objects against a staging meeting.**

**Input:** A set of create and/or update change objects (e.g. from production logs) + a target staging meeting + an **ID map** (when used after Clone).

**What it does:**
- Sanitizes the changes (PII replaced with synthetic values, `isAutomated` stripped from formData)
- **Remaps IDs** in the change objects using the ID map: production change objects reference production item IDs (in `dcId`, `path`, `slag`, `formData`). The ID map from Clone provides a deterministic `source ID → cloned ID` translation — no guessing required.
  - When used standalone (without a prior Clone), falls back to data fingerprinting: pull the meeting, match items by non-ID field values. This is less reliable but allows Apply Changes to work independently.
- Sends create changes first, then update changes (updates may reference records created in the create step)
- Uses the staging meeting's `meetingId` and DCJSON as context for the sync mutations

**Verification:** After syncing, re-pulls the meeting and checks that every change was applied. Produces a report showing which changes succeeded and which didn't, with per-field detail for failures.

**Output:** Sync results + a verification report confirming all changes were applied.

### Capability 3: Replay (composition of Clone + Apply Changes)

**The full production debugging workflow.**

**Input:** Production log files (CreateChanges + UpdateChanges JSON) + a target staging meeting.

**What it does:**
1. **Clone** the staging meeting to match the `initialDCJSON` from the log files (Capability 1)
2. **Apply Changes** from the log files against the cloned meeting (Capability 2), using the **ID map produced by Clone** for deterministic ID remapping

This reproduces the exact production scenario in staging where we can inspect SF records, backend logs, and sync behavior.

## Shared Infrastructure

### Authentication
- Authenticate against the DC backend using email/password login
- Complete Salesforce SSO (OAuth flow with browser-based authorization)
- Handle session expiry and re-authentication
- Support all backend environments (local, dev, UAT, prod, SJP UAT, SJP prod)

### Meeting Search and Pull
- Search for meetings by client name
- Present results for interactive selection when multiple matches exist
- Deduplicate results that share the same Salesforce account ID
- Pull the full DCJSON for the selected meeting
- Detect joint meetings (two clients) from search results
- Apply POA merge transformation after pulling (dcReact's `transformDCJSONForSnapshot`)

### Dump Artifacts
- When a dump directory is specified, save all intermediate JSON at each stage
- Numbered files for clear ordering (01_source, 02_fresh, 03_sanitized, etc.)
- Include verification reports

## Constraints

- **Never send real PII to staging** — all personal information must be replaced with synthetic values before any sync operation
- **Never set PII fields to null** — use synthetic values instead. Null breaks mandatory field validation and causes the diff engine to silently skip items.
- **Creates before updates** — update changes may reference SF record IDs that only exist after creation
- **10-minute timeout** for sync mutations — large meetings with many repeater items need this
- **The tool does not create meetings** — it can only search for and modify existing meetings in the target environment
- **Match dcReact's sync behavior** — the diff engine, change generator, and sanitizer must produce output compatible with what dcReact would produce. The backend expects change objects in a specific format.
- **POA merge transformation** — the DCJSON pulled from SF must have PowerOfAttorney data merged into WillArrangements before diffing, matching dcReact's `transformDCJSONForSnapshot`
- **`isAutomated` must never appear in sync payloads** — SF sets this automatically on records. If present in formData, the backend logs "Missing Mapping" errors and the sync fails.

## Production Log File Format

**CreateChanges file:**
```json
{
  "operation": "mutation",
  "arguments": {
    "meetingId": "uuid",
    "changes": [ ...array of create change objects... ],
    "initialDCJSON": { ...full meeting snapshot... },
    "currentDCJSON": { ...meeting state after local edits... },
    "source": "ios"
  },
  "returns": ["success", "message", "results"]
}
```

**UpdateChanges file:**
```json
{
  "meetingId": "uuid",
  "changes": [ ...array of update change objects... ],
  "initialDCJSON": { ...same snapshot... },
  "currentDCJSON": { ...same state... },
  "source": "ios"
}
```

The `initialDCJSON` in both files is the same: the meeting as it was pulled from production SF before the user made changes.

## CLI Interface

### `dc-sync-probe clone` — Capability 1

```bash
dc-sync-probe clone -e <env> -s "<search term>" --source <dcjson_file.json> [-d <dump-dir>]
```

Clones the source DCJSON into a staging meeting, then verifies.

### `dc-sync-probe apply` — Capability 2

```bash
dc-sync-probe apply -e <env> -s "<search term>" --create <create_changes.json> [--update <update_changes.json>] [--id-map <id_map.json>] [-d <dump-dir>]
```

Applies change objects to a staging meeting, then verifies. When `--id-map` is provided (e.g. from a prior `clone`), uses it for deterministic ID remapping. Without it, falls back to data fingerprinting.

### `dc-sync-probe replay` — Capability 3

```bash
dc-sync-probe replay -e <env> -s "<search term>" --create <create_changes.json> --update <update_changes.json> [-d <dump-dir>]
```

Clone + apply in sequence. Extracts `initialDCJSON` from the log files as the source for cloning. Passes the ID map from clone to apply automatically.

### Utility commands

#### `dc-sync-probe pull` — Pull meeting only

```bash
dc-sync-probe pull -e <env> -s "<search term>" [-o <output.json>]
```

#### `dc-sync-probe diff` — Diff two DCJSONs

```bash
dc-sync-probe diff <original.json> <current.json> [-m <meeting-id>]
```

### Common options

| Option | Description |
|--------|-------------|
| `-e, --env` | Target environment (local, dev, uat, dc-prod, sjp-uat, sjp-prod) |
| `-s, --search` | Client name to search for |
| `-d, --dump-dir` | Directory to save intermediate artifacts |
